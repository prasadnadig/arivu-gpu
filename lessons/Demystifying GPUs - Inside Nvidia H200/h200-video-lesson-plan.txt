NVIDIA H200 Deep Dive: How a Data Center GPU Powers LLMs

This document is an educational reference about the NVIDIA H200 Tensor Core GPU. It explains what the H200 is, how it is architected, and why it is particularly strong for large language models (LLMs) and other AI workloads.

The tone is beginner-friendly. Every acronym is defined on first use.

Chapter 1 – Overview of the H200

The NVIDIA H200 Tensor Core GPU is a data center accelerator based on the NVIDIA Hopper architecture, designed for generative AI, LLMs, and high-performance computing (HPC).

Key headline characteristics from NVIDIA’s H200 datasheets and product pages include:
141 gigabytes (GB) of HBM3e (High Bandwidth Memory 3e).
4.8 terabytes per second (TB/s) of memory bandwidth.
Up to 4 petaFLOPS of FP8 tensor performance.
2× LLM inference performance compared to NVIDIA H100 in certain workloads.

At a high level, the H200 builds on the same Hopper compute architecture as H100, but pairs it with larger and faster HBM3e memory, which is critical for memory-intensive AI workloads.

Chapter 2 – CPU vs GPU vs Data Center GPU

A CPU (Central Processing Unit) is a general-purpose processor optimized for control-heavy tasks, operating systems, and tasks with complex branching. It typically has a small number of powerful cores.

A GPU (Graphics Processing Unit) is a highly parallel processor with many simpler cores optimized for numerical workloads such as graphics, simulations, and matrix math.

A data center GPU like the H200 is distinct from consumer GPUs in that it:
Is designed for continuous 24/7 operation in servers.
Prioritizes throughput, memory capacity, and multi-GPU scaling rather than display output.
Integrates into server platforms rather than desktops.

Chapter 3 – High-Level H200 Specifications

According to NVIDIA H200 datasheets and partner datasheets, typical key specifications include:
Architecture: NVIDIA Hopper.
GPU memory: 141 GB HBM3e.
Memory bandwidth: 4.8 TB/s.
FP8 Tensor Core performance: up to 4 petaFLOPS.
FP16/BF16 Tensor Core performance: up to ~1,979 TFLOPS.
FP32 performance: ~67 TFLOPS.
FP64 performance: ~34 TFLOPS.
Interconnect: NVLink (up to 900 GB/s GPU–GPU bandwidth) and PCIe Gen5.
Form factors: SXM and NVL variants, depending on system configuration.
TDP (Thermal Design Power): up to around 600–700 W for some SXM/NVL configurations.

Compared to NVIDIA H100, the H200:
Increases memory from 80 GB to 141 GB.
Increases memory bandwidth from ~3.35 TB/s to 4.8 TB/s.
Keeps similar FP32/FP64 compute, but leverages larger and faster HBM to improve LLM throughput.

Chapter 4 – Chip, Package, and Module

The H200 can be understood at three levels:
GPU die: the silicon chip containing Streaming Multiprocessors (SMs), caches, Tensor Cores, and control logic.
Package: the assembly that mounts the GPU die together with multiple HBM3e stacks and provides electrical connections.
Module: the complete product that plugs into a server board.

In the H200, the GPU die and several HBM3e stacks sit side by side on the same package, connected by very wide memory interfaces. The module adds power delivery, signaling, and cooling integration.

Chapter 5 – H200 Architecture and Memory Subsystem

5.1 SMs, L2 Cache, and HBM3e

In Hopper-based GPUs like H200, compute units are grouped into Streaming Multiprocessors (SMs). These SMs are organized into larger clusters and share a large L2 cache, which sits between the SMs and the HBM3e memory stacks.

The H200’s memory subsystem features:
Multiple HBM3e stacks totaling 141 GB.
Aggregate memory bandwidth of 4.8 TB/s.
An L2 cache designed to buffer data and reduce repeated trips to HBM3e.

5.2 Memory Hierarchy

The memory hierarchy inside H200 can be summarized as:
Registers: per-thread, fastest and smallest.
Shared memory / L1: per-SM scratchpad used for inter-thread data sharing and caching.
L2 cache: shared across the GPU.
HBM3e: large off-core memory on the package.

Effective kernels aim to load data from HBM3e into L2 and shared memory once, then reuse it from registers and shared memory to minimize off-chip traffic.

Chapter 6 – Streaming Multiprocessors and Execution Model

6.1 Streaming Multiprocessors (SMs)

Each H200 SM contains:
4th-generation Tensor Cores for matrix operations.
Scalar and vector ALUs for general arithmetic.
Warp schedulers that issue instructions to groups of threads.
Registers and shared memory/L1.

SMs are designed to run many warps concurrently, switching between them to hide memory latency and keep the compute units busy.

6.2 Threads, Warps, and SIMT

H200 follows NVIDIA’s SIMT (Single Instruction, Multiple Threads) execution model:
A thread is the smallest execution unit.
Threads are grouped into thread blocks, which share shared memory and can synchronize.
A warp is a group of 32 threads that execute the same instruction together.

The SM’s warp schedulers issue instructions to warps, and each thread in the warp performs that instruction on its own data. This model is ideal for data-parallel operations such as matrix multiplication.

Chapter 7 – Precision Modes and Tensor Performance

H200 supports multiple precision types via Hopper Tensor Cores:
FP8 (8-bit floating point) for high-throughput AI, especially inference.
FP16 / BF16 (16-bit floating points) for training and inference.
TF32 / FP32 for workloads requiring more precision.
FP64 for HPC and scientific computing.

Representative Tensor Core performance for Hopper-class GPUs such as H200 includes:
FP8 Tensor Core: up to ~3,958 TFLOPS.
FP16/BF16 Tensor Core: up to ~1,979 TFLOPS.
FP32 Tensor Core (TF32): up to ~989 TFLOPS.
FP32 and FP64 base FP performance similar to H100.

The H200 combines this tensor performance with larger and faster HBM3e, enabling higher effective throughput for large AI models.

Chapter 8 – NVLink and Multi-GPU Scaling

NVLink is NVIDIA’s high-speed GPU–GPU interconnect. H200 supports NVLink 4, offering up to about 900 GB/s GPU–GPU bandwidth per GPU in certain NVL configurations.

Key properties:
Enables high-bandwidth, low-latency communication between GPUs.
Supports 2-way and 4-way NVL configurations and larger NVLink/NVSwitch systems.
Complements PCIe Gen5, which is available for CPU–GPU connectivity.

For LLMs and large models, NVLink is crucial for:
Model parallelism.
Gradient synchronization in data-parallel training.
Handling long context windows and large batch sizes without saturating PCIe.

Chapter 9 – Why H200 Excels at LLMs

The H200’s design choices map directly onto the needs of LLMs:
Memory Capacity (141 GB HBM3e)
This allows single GPUs to hold larger models, such as 70B-parameter LLMs at low precision, reducing or eliminating the need to shard parameters across many GPUs.
Memory Bandwidth (4.8 TB/s)
High bandwidth keeps Tensor Cores supplied with data during large matrix operations, particularly important for attention and feed-forward layers.
Tensor Performance
FP8 and FP16/BF16 modes provide very high throughput, enabling fast training and inference.
NVLink Scaling
NVLink 4 with up to 900 GB/s bandwidth allows multiple H200s to work together efficiently in large systems.

NVIDIA and partner materials report up to roughly 2× LLM inference performance improvement over H100 NVL in certain configurations, thanks primarily to the H200’s larger and faster memory subsystem.

Chapter 10 – GEMM (General Matrix Multiply) on H200

10.1 GEMM as the Core Operation

Most of the heavy computation in LLMs boils down to GEMM (General Matrix Multiply) operations:
Attention projections.
Feed-forward layers.
Gradient computations during training.

These all reduce to multiplying large matrices and adding results.

10.2 Tiling and Parallel Execution

On H200, large matrix multiplications are tiled:
Input matrices are broken into smaller tiles.
Each tile is assigned to a thread block, which runs on an SM.
Each block loads tiles of input matrices from HBM3e into shared memory and reuses them for multiple multiply-add operations.
Many tiles are processed in parallel across multiple SMs.

10.3 Data Flow Inside the SM

Within an SM handling a GEMM tile:
Load/store units fetch matrix tiles from HBM3e via L2 cache into shared memory.
Warps load portions of these tiles from shared memory into registers.
Tensor Cores perform fused multiply-add operations on register data.
Partial results accumulate in registers and are periodically stored back to shared memory or global memory.

This design maximizes reuse of data that is already on-chip and minimizes expensive trips to HBM3e.

Chapter 11 – Backpropagation and Training on H200

Backpropagation is the process used to train neural networks by computing gradients of a loss function with respect to model parameters.

The forward pass computes outputs and stores intermediate activations.
The backward pass computes gradients by applying the chain rule, often using more GEMMs and elementwise operations.

On H200:
The same GEMM infrastructure used for forward passes accelerates backward passes.
Many layers’ forward and backward GEMMs can be pipelined and parallelized across SMs.
Large models benefit from H200’s 141 GB HBM3e, which can hold parameters, activations, and optimizer states with fewer data transfers.

Chapter 12 – Example LLM Training Scenario

Consider training or fine-tuning a large LLM:
Parameters and optimizer states are stored in HBM3e across one or more H200 GPUs.
Input sequences and batches are distributed across SMs.
GEMM-heavy layers are executed in parallel across SMs and, if needed, across multiple H200s connected by NVLink.

The large HBM3e capacity simplifies model parallelism and reduces the need for complex parameter and activation offloading strategies, especially at low precision.

Chapter 13 – CPU and H200 Roles in a System

In a typical server:
The CPU manages:
Data loading and preprocessing.
High-level orchestration of training or inference jobs.
Operating system tasks.

The H200 GPU performs:
Large-scale matrix multiplications.
Tensor operations for deep learning.
Parallel numeric kernels.

Together, the CPU and H200 form a heterogeneous system where each component focuses on tasks suited to its strengths.

Chapter 14 – Mental Model and Summary

To build an intuitive mental model of H200:
Think of it as a Hopper-based GPU with:
141 GB of HBM3e memory.
4.8 TB/s of memory bandwidth.
Powerful Tensor Cores for low-precision matrix math.

Its memory hierarchy is designed to reuse data and keep Tensor Cores busy.
Its SIMT execution provides massive parallelism.
Its NVLink capabilities allow multiple H200s to behave as a coherent, high-bandwidth compute fabric.

From an LLM perspective, H200 is best thought of as a massively parallel matrix engine with large, fast memory, carefully engineered to hold and process large models efficiently.
